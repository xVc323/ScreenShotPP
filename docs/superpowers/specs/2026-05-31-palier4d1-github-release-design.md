# Palier 4d.1 — Public GitHub Release — Design

- **Date**: 2026-05-31
- **Status**: Approved design, pending implementation plan
- **Depends on**: functional macOS and Windows builds; green CI on `master`
- **Parent context**: `docs/superpowers/specs/2026-05-30-screenshotpp-design.md` (§2 bis, distribution)

---

## 1. Goal

Publish ScreenShotPP as a polished open-source project on GitHub with a first downloadable
release:

- a signed and notarized macOS `.dmg`;
- a Windows NSIS `.exe`;
- a public repository that presents the product clearly to non-developers;
- an automated tag-driven release workflow.

The GitHub release is the first distribution channel. Mac App Store and Microsoft Store
publication remain separate follow-up milestones.

---

## 2. Product and pricing strategy

ScreenShotPP remains fully open source under the MIT license.

Distribution channels:

| Channel | Availability | Price |
|---|---|---|
| GitHub Releases | First milestone | Free |
| Mac App Store | Follow-up milestone | EUR 2.99 |
| Microsoft Store | Follow-up milestone | EUR 2.99 |

All channels provide the same application features. There is no donation system, paywall,
or in-app purchase in v1. Store pricing compensates for installation convenience,
automatic updates, discovery, and user trust rather than unlocking functionality.

## 4. First release assets

The first public release is tagged `v0.1.0`.

Published assets:

| Platform | Asset | Status |
|---|---|---|
| macOS | `ScreenShotPP_0.1.0_aarch64.dmg` | Signed with Developer ID Application, notarized, stapled |
| Windows | NSIS `.exe` installer generated by Tauri | Preview: unsigned initially |

The Windows installer is published because Windows is a core product target and the
original motivation for the project. Until Windows code signing is available, release
notes and the README explicitly state that Microsoft SmartScreen may show a warning.

After the repository and first release become public, apply to SignPath Foundation for
free open-source Windows signing. Integrate signed Windows releases in a future `v0.1.1`.

Portable archives, `.msi`, auto-update feeds, Homebrew, and Winget are out of scope for
this milestone.

## 5. Automated release workflow

A GitHub Actions workflow runs on tags matching `v*`.

### macOS job

1. check out the tagged source;
2. install the Rust and Node toolchains;
3. import the Developer ID Application certificate into a temporary keychain;
4. build the macOS bundle through Tauri;
5. sign the `.app` and `.dmg`;
6. submit the `.dmg` to Apple notarization;
7. wait for the result;
8. staple the notarization ticket;
9. validate the stapled `.dmg`;
10. upload the `.dmg` to the GitHub Release.

### Windows job

1. check out the tagged source;
2. install the Rust and Node toolchains;
3. build the NSIS `.exe` through Tauri;
4. upload the installer to the GitHub Release.

### Release publication

The workflow creates the GitHub Release automatically from the tag. Release notes include:

- concise feature highlights;
- macOS installation;
- Windows installation;
- an explicit SmartScreen preview note for the unsigned Windows installer;
- a link to the source code and MIT license.

The workflow fails rather than publishing incomplete assets if either required platform
job fails.

## 6. Secrets and credentials

Apple distribution secrets live only in GitHub Actions encrypted secrets.

Expected secrets:

- Developer ID Application certificate exported as base64-encoded `.p12`;
- `.p12` password;
- temporary keychain password;
- Apple Team ID;
- App Store Connect API key identifier;
- App Store Connect issuer identifier;
- App Store Connect API private key.

Exact secret names are selected during implementation and documented in a maintainer-only
setup guide. Secrets must never appear in Git, README content, generated release notes, or
workflow logs.

The local developer identity `ScreenShotPP Dev` remains useful for development builds but
must not sign public releases.

## 7. Verification

Before changing repository visibility:

1. CI is green on `master`;
2. the README renders correctly on GitHub;
3. the GIF loads and demonstrates the complete workflow;
4. the MIT license is detected by GitHub;
5. a release-candidate tag validates the release workflow while the repository is still
   private;
6. the downloaded `.dmg` passes Apple notarization and stapling validation;
7. the installed macOS app launches and completes a capture;
8. the downloaded Windows `.exe` installs and completes a capture on Windows;
9. the release notes clearly identify the unsigned Windows preview.

After those checks, make the repository public and publish `v0.1.0`.
